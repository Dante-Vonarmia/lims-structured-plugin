# 模板策略与字段模型

更新时间：2026-03-29

## 结论

当前系统不是“每个模板一套字段池”。  
采用的是“统一字段模型 + 模板画像 + 通用语义填充 + 少量历史特例”的分层策略。

## 1. 字段池策略

### 1.1 全局主字段池（共享）

所有模板共享一套主字段（示例）：

- `device_name`
- `manufacturer`
- `device_model`
- `device_code`
- `certificate_no`
- `basis_standard`
- `location`
- `temperature`
- `humidity`
- `general_check_full`

这些字段是跨模板复用的数据模型，不按模板拆分。

### 1.2 字段池扩展原则

只有满足以下任一条件，才建议升格进字段池：

1. 需要单独检索/统计
2. 需要规则校验（跨记录/跨模板）
3. 需要跨模板稳定复用

对于“倍率/标准值/实际值”这类明细矩阵，优先作为结构化表格处理，不拆成大量离散字段。

## 2. 模板画像（Profile）策略

模板画像用于描述“编辑层与映射层”差异，不是独立字段池：

- 编辑页面展示哪些字段
- 字段顺序/分组/宽窄
- 个别模板补充的映射提示

画像配置位于 `backend/app/rules/template_profiles/`。

## 3. 通用填充策略

生成时优先使用通用语义填充，不依赖模板编号：

1. 通过表头语义识别键列和值列  
   示例语义：`标准值/标称值`、`实际值/实测值`
2. 用键列匹配行，再回填值列
3. 对常见文本占位（如 `扩展不确定度U=...`）按语义回填

该策略目标是“同类结构可复用”，而非“每个模板独立写一套逻辑”。

## 4. 模板专用规则边界

允许存在少量模板专用规则，但应满足：

1. 先尝试通用语义方案
2. 专用规则只用于通用语义无法稳定覆盖的场景
3. 专用规则不影响通用主回路

## 5. 识别与生成链路（简版）

1. 识别阶段提取原始字段与结构化块（含 `general_check_full`）
2. 编辑阶段可视化修改字段与表格结构
3. 生成阶段统一走 `render_report` -> `fill_generic_record_docx` / 固定处理器
4. 优先通用语义填充，再执行必要特定规则

## 6. 当前实践建议

1. 继续增强“结构化表格能力（matrix）”
2. 控制字段池规模，保持主字段稳定
3. 新模板先做“语义可对齐”验证，再决定是否加特例
4. 对同类表格优先复用通用表头语义匹配能力
